＋為什麼有「Mock / 真模型」兩套？架構

每個重模型都切成一個乾淨介面，底下掛兩種實作：Mock 版（免下載、用傳統演算法，讓流程立刻能跑）和 真實版（接真模型）。用環境變數一鍵切換。

好處：全隊 Day 1 就能對著會動的系統開發，不用等模型；換模型時上層 API、前端、分類器完全不用改。這一週做的就是把 SAM 這一格從 Mock 換成真的。

MockSegmenter（OpenCV，現在） ⇄ SamSegmenter（真 MobileSAM，本週補）
＋什麼是主動學習迴圈？核心賣點

一般標註是人把每張圖都標一遍，很花時間。主動學習反過來：讓模型先自動標，只把「模型自己也沒把握（信心分數低）」的少數樣本挑出來給人審核。

人只修正這些關鍵樣本 → 餵回模型 → 模型變準 → 下一輪不確定的更少。這個自動標 → 挑低信心 → 人工修正 → 回饋重訓的循環，就是主動學習迴圈。在我們系統裡就是「工程師只審核標紅的樣本，其餘全自動」。

自動標 → 挑低信心（標紅）→ 人工修正 → 回訓 → 更準 →（循環）
＋為什麼定案用 MobileSAM？已定案

原版 SAM 又大又慢（模型好幾 GB），不適合放上線，所以要用輕量版。常見兩個選擇，我們已定案 MobileSAM：

版本做法特點
MobileSAM ←採用把 SAM 的影像編碼器換成輕量骨幹（vit_t）用法幾乎跟原版 SAM 一樣，遷移最省事
FastSAM改用 YOLOv8 架構做全圖分割更快，但 API 跟 SAM 不同，接法要另寫

關鍵理由：SamSegmenter 的骨架已照原版 SAM 的用法寫好（sam_model_registry / SamAutomaticMaskGenerator / SamPredictor）， MobileSAM 的 API 幾乎照搬，峻維只要把權重換成 vit_t 版就能接上，改動最小、風險最低。FastSAM 若日後想比較速度，可當延伸實驗。

＋為什麼是 768 維向量？DINOv2

DINOv2 的工作是把一張圖壓成一串數字來代表內容，這串數字有多長就叫幾維。維度越高、資訊越豐富、越能分辨細節，但越吃運算。

768 不是我們挑的，是「dinov2-base」這個版本架構寫死的輸出長度（改不了）。現在的 Mock 版只有 50 維（顏色+形狀），只能靠顏色分；768 維的 DINOv2 才真的「看懂」這是貓那是狗。

版本維度特點
dinov2-small384最小最快
dinov2-base ←目前選768平衡
dinov2-large1024更準更重

關鍵：換模型時分類器完全不用改——它是照向量長度動態算的，餵 50 維或 768 維都吃。若 base 太重跑不動，退用 small（384）只要改兩行。
