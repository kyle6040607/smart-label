四週鋪墊（Dockerfile、compose、LINE、GCP 帳號）都是為了這一刻：把系統放上網。 先求通、再求快——先用最簡單的路徑上線，優化留給有餘裕的時候。

① 上雲（7/27 → 7/29，關鍵路徑）

  • 已備：Dockerfile、docker-compose（含 MySQL）、GCP 帳號與專案
  • 開一台 Compute Engine VM（e2-standard-4 級即可，CPU 推論；用峻維的 CPU 耗時表估規格）→ docker compose up -d 整組帶起來
  • HTTPS：LINE webhook 強制要求。最省事的路徑是 Caddy / nginx + Let's Encrypt 反向代理，搭一個免費子網域（如 DuckDNS）；別為了網域卡兩天
  • 環境變數用 prod 值（跟佳凱拿 SECRET_KEY、DB 密碼）；上傳目錄掛 volume，重開容器資料不掉
  • 雲端跑的是 CPU：真模型參數用峻維調過的提速組（縮圖 + 少點），先實測素人流程單張耗時，超過 15 秒就再降參數——雲端 demo 以素人流程（框提示，天生快）為主

② LINE 正式化 + 穩定性（7/30 → 7/31）

  • LINE Developers 後台把 webhook 從 ngrok 切到正式網址，完整走一次「加好友 → 選單上傳 → 打 cat → 收結果圖」

  • 觀察兩天穩定性：記憶體夠不夠（真模型 + MySQL 同機）、掛了會不會自動重啟（compose restart: unless-stopped）

名詞小百科（講給組員與評審聽）

＋雲端沒有 GPU，模型跑得動嗎？部署現實

跑得動，但要選對路徑。GPU 雲主機貴（時租數十倍於 CPU 機），學生專題沒必要。我們的對策有三層：

① 素人流程天生快——Grounding DINO 出框、SAM 吃框提示，只算一兩個點，CPU 上幾秒可接受；最慢的「整張自動分割」是工程師功能，demo 不靠它。 ② Week 3 的提速調參（縮圖、減提示點）就是為 CPU 部署準備的。 ③ 重訓練（YOLO）本來就不在伺服器上跑，在 Colab 的免費 GPU 跑完把結果帶回來。

雲端 CPU：素人打字流（秒級）✅　整張自動分割（慢，demo 不用）⚠️　YOLO 訓練（丟 Colab GPU）✅
＋mAP 是什麼？多少算好？成果指標

mAP（mean Average Precision）是物件偵測/分割的標準成績單：模型框（切）得準不準、找得全不全，綜合成 0～1 的分數。mAP50 是「跟正解重疊過半就算對」的寬鬆版，mAP50-95 是嚴格版。

我們的重點不是把 mAP 衝到頂（資料量小、類別少），而是講這個故事：用本平台「幾乎不用人工」標出的資料，訓練出的模型達到 X mAP——證明整條「標註 → 資料集 → 模型」生產線是真的。誠實秀一張失敗案例並解釋原因，比灌水數字更加分。

＋為什麼要三層保險？demo 風險

發表現場的定律：會壞的都挑那天壞——教室 Wi-Fi 慢、雲端剛好重啟、LINE API 抽風。所以逐層降級，每一層事先真的演練過：

層方案啟用時機
1雲端公開網址（主）預設
2發表筆電上的本機 docker-compose雲端或網路掛
3預錄 demo 影片連本機都出狀況

關鍵是切換要果斷：現場卡超過 20 秒就降一層，別讓評審看你重整頁面三次。
