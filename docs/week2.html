＋兩種模式 × 兩種操作，到底差在哪？核心賣點

我們的產品服務兩種人，所以有兩種操作方式：

自然語言操作——打一句話（「cat」）系統就幫你框出物件、切好。門檻最低，素人和 LINE 只能用這個。背後是 Grounding DINO。

種子樣本操作——人手動描邊 / 點選幾個範例當「種子」，讓 few-shot 分類器學會你的專屬類別。比較費工但更精準可控，只開放給工程師模式。

模式種子樣本自然語言調增強參數
素人 / LINE✗✓✗
工程師✓✓✓

關鍵：這道權限線要放後端，不是前端把按鈕藏起來就算數——素人的 API 根本打不到種子樣本 / 參數端點。

＋Grounding DINO 是什麼？跟 SAM 怎麼合作？自然語言引擎

SAM 很會「切」，但它不知道你想要哪一塊，所以要嘛整張自動切、要嘛手動點。Grounding DINO 補的正是這塊：它是 open-vocabulary 偵測器，給一句自由文字（"cat"），它就框出畫面裡對應的物件。

兩者串起來就是業界很紅的 Grounded-SAM：文字 → Grounding DINO 出框 → 框餵給 SAM → 精準遮罩。使用者不必手動點，打字就能標——這正是我們素人模式的操作方式。SAM 的 predictor 上週已寫好，直接吃 box prompt。

"cat" → Grounding DINO（框）→ SamPredictor.predict(box=…) → MaskDict（照舊）→ 分類 / 審核 / 存 MySQL
＋為什麼把 store.json 換成 MySQL？資料庫

骨架期用一個 store.json 檔存全部資料，開發方便，但一旦要多使用者、雲端儲存、給每個人一塊影像空間，單一 JSON 檔就撐不住——併發會壞、也難查詢關聯。

所以升級成 MySQL：使用者、影像、片段、種子範例各自有表，用外鍵關聯（「這張圖屬於哪個使用者、切出哪些片段」）。好處是併發安全、查詢快、天生支援「使用者 ↔ 影像雲端空間」這種關係。

關鍵：靠佳凱定的儲存層抽象，Repository 介面不變，只是底層從讀寫 JSON 換成讀寫 MySQL——pipeline 和 routes 一行都不用改。

store.json（單檔，開發用） → MySQL（多使用者 · 影像歸屬使用者 · 關聯查詢）
＋LINE Bot 怎麼變成「操作模型」的入口？素人入口

對素人來說，裝一個網站、學一套介面都嫌麻煩。LINE 是他們每天都在用的東西，所以我們把模型接進 LINE Bot：使用者傳一張圖、打一句「cat」，Bot 就回傳切好的結果。

技術上，LINE 的 webhook 收到訊息 → 後端呼叫和網頁同一個 segment_text（就是峻維做的自然語言引擎）→ 把結果圖回給使用者，並把圖歸到該 LINE 使用者名下。LINE 一律走素人模式，只認自然語言，不開放種子樣本那些進階操作。

LINE 傳圖 + 「cat」 → webhook → segment_text（自然語言引擎）→ 回結果圖 + 存 MySQL（歸該使用者）
